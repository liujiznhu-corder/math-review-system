# 学生端 UI 设计规范

本文档用于指导“江苏专转本数学错题复盘系统”学生端后续 UI 改造。项目技术栈为 Next.js App Router、TypeScript、Tailwind CSS、Supabase。教师端暂不纳入本轮 UI 美化范围。

## 1. 设计定位

学生端不是后台管理系统，而是学生每天打开的 Web 学习工具。实际使用场景以笔记本电脑、iPad / 平板浏览器为主，手机为补充场景。

设计参考：

- Notion 的轻量卡片和信息分组。
- Anki 的复习节奏、任务感和掌握反馈。
- DaysMatter 的倒计时仪式感。
- 现代 Web App 的清晰层级、舒适留白和稳定交互。

整体目标是让学生快速判断“今天要做什么、从哪里开始、完成后有什么反馈”，而不是展示给老师或管理员看的数据后台。

## 2. 布局与宽度

学生端应采用桌面端 / iPad / 平板优先，兼容手机端的策略。

- 保留现有网页端舒适宽度，不强制使用 `max-w-[480px]`。
- 主页面可继续使用 `max-w-5xl`、`max-w-6xl` 或现有舒适宽度。
- 详情页、答题页可使用更聚焦的 `max-w-4xl`。
- 桌面端允许两栏布局，例如主内容 + 侧栏提示。
- 平板端保持宽内容卡片和清晰分区。
- 手机端需要响应式折叠为单列，但不是唯一设计基准。
- 所有宽度下都必须避免横向溢出，长公式区域可单独横向滚动。

不要把学生端改成手机小程序窄屏，也不要做成复杂数据大屏或后台管理页。

## 3. 整体风格

- 浅色背景，轻卡片，柔和阴影，清晰层级。
- 页面区块要像学习任务面板，而不是数据表。
- 卡片用于承载任务、题目、统计和入口，不要堆过多装饰。
- 按钮文案要具体，直接说明动作结果。
- 保持当前网页端导航和核心入口，不为了美化删除流程。

推荐背景与主色：

- 背景：`bg-slate-50`、`bg-[#F7F8FC]`、浅蓝灰背景。
- 主色：blue / indigo / sky / emerald。
- 成功、掌握、完成状态可用 emerald。
- 待复习、进行中、推荐训练可用 sky 或 indigo。
- 警示色只在必要时使用 red，避免传统教育软件式大红大蓝。

## 4. 通用组件建议

后续可逐步抽取或复用：

- `StudentPageShell`：学生端统一页面容器，支持 `max-w-6xl`、`max-w-5xl`、`max-w-4xl` 等宽度变体。
- `StudentHeader`：页面标题、说明、返回入口、状态提示。
- `StatCard`：倒计时、今日任务数、完成进度、连续复习天数。
- `ActionCard`：录入错题、开始复习、开始训练等主入口。
- `ProblemCard`：错题或训练题卡片，展示题型、预览、状态和操作。
- `LatexProblemBlock`：完整题目展示区，长公式允许横向滚动。
- `PrimaryButton` / `SecondaryButton`：统一按钮视觉与 loading / disabled 状态。
- `EmptyState`：无数据状态，给出下一步行动。
- `LoadingSkeleton`：页面或卡片加载骨架。

组件抽取要服务现有页面，不要为了组件化重写业务逻辑。

## 5. 首页 / Dashboard

当前 `/dashboard` 已包含考试倒计时、快捷入口、今日复盘状态、薄弱题型、知识点掌握度和最近 7 天学习总结。后续 UI 改造应保留这些功能，只提升视觉层级和学习仪式感。

首页应优先回答：

- 距离考试还有多久。
- 今天最应该完成什么。
- 我现在可以从哪里开始。
- 哪些题型最需要突破。

必须保留的入口：

- 录入错题。
- 错题库。
- 今日复习。
- 薄弱巩固。
- 专项训练。

注意：`/dashboard` 当前同时包含教师分支，学生端 UI 改造不能影响教师 dashboard。

## 6. 错题录入

`/mistakes/new` 是学生端核心入口，后续 UI 改造不能删除或弱化任何关键能力。

必须保留：

- LaTeX 输入框，当前字段为 `latexContent`。
- LaTeX 实时预览。
- AI 录题助手入口。
- 智能推荐题型按钮。
- 推荐 top 3 的展示与选择能力。
- 手动选择题型。
- 备注输入。
- “保存错题”按钮。
- “提交教师审核”按钮。
- 学生不确定题型时，可以不选题型直接提交教师审核。

业务约束：

- 提交教师审核时，题型可为空，分类状态为待审核。
- 提交教师审核后不生成复习任务。
- 选择题型并保存后进入复习计划。
- 不接入新的 OCR / AI API，不把密钥暴露到客户端。

UI 方向：

- 保留当前左右布局：主录入区 + 推荐说明 / 推荐结果区。
- 强化“输入 LaTeX -> 预览 -> 推荐/选择题型 -> 保存或审核”的流程感。
- 推荐和保存按钮都必须有 loading / disabled 状态。

## 7. 今日复习

`/reviews` 当前展示今日待复习任务、题号导航、当前题题干、复习轮次、查看答案、已掌握、未掌握、上一题、下一题。

必须保留：

- 今日待复习数量。
- 今日已完成数量。
- 当前进度。
- 题号导航。
- 错题题干与 LaTeX 渲染。
- 复习轮次，例如第 1 天、第 3 天、第 7 天、补复习第 3 天。
- “查看答案”。
- “已掌握”。
- “未掌握”。
- 上一题 / 下一题。

业务约束：

- 点击“未掌握”后，现有逻辑会追加补复习任务。
- 不要把数据库字段名直接暴露给学生，例如 `day1`、`retry_day3`，应转换为可理解文案。

UI 方向：

- 任务多时，题号导航需要更清晰的完成 / 未完成状态。
- 当前题区域应更聚焦，避免题号列表压过题目主体。
- 答案和解析仍应在点击后展示，不要首屏全部展开。

## 8. 错题库

`/mistakes` 当前是卡片列表，不是表格。页面支持题型筛选、分类状态筛选、关键词搜索、分页和查看答案。

必须保留：

- 只显示当前学生有权限看到的错题。
- 一级 / 二级 / 三级题型筛选。
- 分类状态筛选。
- 关键词搜索。
- 分页与 page size。
- 分类状态，例如待教师审核、学生已选择、教师已确认。
- 录入时间。
- 查看答案入口。

展示建议：

- 错题库要像学生自己的错题本，不要做成数据表。
- 默认列表只展示题目预览、题型、状态、日期和操作。
- 完整题目、答案、解析放到详情页。
- 当前列表会渲染较多 LaTeX，后续优化应控制列表页公式数量，必要时改为简短预览。

## 9. 专项训练

`/practice` 当前是专项训练入口，支持一级 / 二级 / 三级题型选择，每次固定 5 题，并通过“开始训练”进入训练流程。

必须保留：

- 三级题型选择。
- 每组默认 5 题。
- 开始训练。
- 题量不足时按现有逻辑补足相近题。
- 训练题、答案解析、掌握 / 未掌握、加入错题库等既有流程。

性能要求：

- 首屏不要加载全部题目。
- 默认只取 5 题。
- 不要首屏加载全部答案、解析、错题、复习记录。
- 不要为了随机 5 题拉取全题库。
- 答案解析应按需加载。
- 训练页只渲染当前题的完整 LaTeX。

UI 方向：

- 专项训练要像“训练入口”，不是普通题目列表。
- 突出题型选择、每组 5 题、开始行动。
- loading、disabled 和错误提示要明确。

## 10. 错题详情

`/mistakes/[id]/answer` 当前用于查看题目、答案解析和返回错题库。

必须保留：

- 返回上一页。
- 题型与分类状态。
- 题干 LaTeX 渲染。
- 答案解析展示。
- 教师备注或待补充状态。

UI 方向：

- 详情页可使用 `max-w-4xl`，保持阅读聚焦。
- 题目和答案解析分区要清楚。
- 长公式区域必须允许横向滚动。

## 11. LaTeX 展示规范

数学题是系统核心，LaTeX 展示要优先保证可读性和性能。

- 完整题目放在题卡、答题页或详情页。
- 长公式允许横向滚动，不撑破页面。
- 列表页只展示必要预览，避免大量同步 KaTeX 渲染。
- 错题录入页必须保留实时预览。
- 答案和解析尽量按需展示。
- 不要为了视觉统一隐藏公式横向滚动能力。

## 12. 交互规范

按钮文案应具体，例如：

- 开始今日复习。
- 查看答案。
- 已掌握。
- 未掌握。
- 开始训练。
- 再练一组 5 题。
- 保存错题。
- 提交教师审核。

异步操作要求：

- 所有提交、推荐、切题、加载更多、开始训练都必须有 loading 或 disabled 状态。
- 避免重复提交。
- 页面级加载应提供 `loading.tsx` 或骨架屏。
- 错误提示要告诉学生下一步怎么做。

## 13. 开发约束

后续学生端 UI 改造必须遵守：

- 不修改数据库结构。
- 不影响 Supabase RLS。
- 不暴露 `SUPABASE_SERVICE_ROLE_KEY` 到客户端。
- 不影响教师端页面。
- 不删除现有功能。
- 不重写业务逻辑。
- 不引入复杂 UI 库。
- 不为了 UI 美化牺牲性能。
- 每次只改 1 个阶段或 1-2 个页面。
- 列表查询必须有合理 limit / pagination。
- 首屏不加载全量题库、全量错题、全部答案解析。
- 修改后必须通过 `npm.cmd run lint` 和 `npm.cmd run build`。
- UI 开发只在 `main` 分支进行。
- `production` 分支只保留稳定上线版本。
- 香港服务器只部署 `production` 分支。
- 不要直接把未经测试的 UI 改动部署到生产环境。

## 14. UI 改造自测要求

每次 UI 改造都必须先建立页面级小基线，再修改，再回归检查。

- 改造前，先记录要修改页面的现有功能、核心按钮和关键业务流程。
- 改造后，必须用测试 student 账号登录本地系统检查页面。
- 检查核心功能是否保留，例如 LaTeX 预览、保存错题、提交教师审核、查看答案、已掌握、未掌握、开始训练等。
- 检查桌面端、iPad 宽度和手机宽度下是否无明显布局错乱或横向溢出。
- 修改后必须运行 `npm.cmd run lint` 和 `npm.cmd run build`。

## 15. 改造顺序

第一阶段：首页 dashboard + 学生端通用视觉基础。

- 建立 `StudentPageShell`、基础卡片、按钮、空状态、骨架屏等。
- 只优化学生 dashboard，不影响教师 dashboard。

第二阶段：错题录入。

- 保留 LaTeX 输入、实时预览、AI 录题助手、智能推荐、手动选题型、保存错题、提交教师审核。
- 优化流程层级和主次按钮。

第三阶段：今日复习。

- 优化任务进度、题号导航、当前题聚焦和复习轮次文案。

第四阶段：错题库。

- 优化错题本卡片、筛选区、分页和公式预览性能。

第五阶段：专项训练。

- 优化训练入口、题型选择、5 题一组和训练过程 loading 体验。
